# 需求背景与目标
## 项目概述
构建一个 AI 驱动的面试模拟系统（Interview Simulator），首版以文本面试为主，重点解决“不会答、答不深、答不稳、答完不知道怎么改”的问题。

核心能力：
- 模拟真实面试流程，而不是做题式问答。
- 基于简历、岗位与目标方向生成个性化问题。
- 根据用户回答进行多轮追问，验证理解深度。
- 输出结构化面试报告，帮助用户复盘与二次练习。

## 1.0 核心目标
- 用户可以在 3 分钟内完成配置并开始一场模拟面试。
- 用户可以完成一场 15-30 分钟的文本模拟面试。
- 系统可以输出包含评分、问题拆解和改进建议的结构化报告。
- 系统可以基于本次面试结果给出下一轮练习方向。

## 建议跟踪的成功指标
- 面试开始率：创建面试后成功进入首题的比例。
- 面试完成率：开始面试后完成整场模拟的比例。
- 报告查看率：生成报告后查看完整报告的比例。
- 用户主观有用性：用户对“问题是否像真实面试”“反馈是否有帮助”的评分。
- 7 日复训率：用户在 7 天内再次开启训练的比例。

## 核心问题
### 目标用户画像
🎯 第一阶段建议聚焦：
- 应届生和 1-3 年经验的求职者。
- 技术岗用户，优先支持 Java 后端与 AI 应用开发方向。
- 有明确求职意向、愿意主动训练并接受反馈的用户。

说明：原文中出现了“前端”，但整体定位更偏后端 / AI 岗，建议 1.0 不同时覆盖过多岗位，以免题库质量和追问深度失控。

### 典型使用场景
| 场景 | 用户目标 |
| --- | --- |
| 面试前 | 模拟真实流程，降低紧张感。 |
| 面试失败后 | 复盘问题，定位薄弱点。 |
| 日常训练 | 提升表达、结构化回答和项目讲述能力。 |
| 简历准备 | 提前暴露项目细节漏洞。 |

### 核心痛点
- 不知道面试官会从哪里切入提问。
- 会知识点，但无法组织成有逻辑的回答。
- 项目经历一旦被追问就容易暴露理解不深。
- 缺少客观、连续、可执行的反馈来源。
- 无法长期积累自己的薄弱题型和成长轨迹。

### 核心价值主张
让用户“像在真的面试里一样”接受一位懂简历、会追问、能复盘的 AI 面试官训练。

### 用户故事
作为一名正在求职的技术岗候选人，我希望在正式面试前进行接近真实场景的模拟练习，让系统根据我的简历、目标岗位和回答内容不断追问，并在结束后告诉我回答中的问题、原因和改进方式，从而更有针对性地准备下一次面试。

## 产品原则
- 真实感优先于题量：宁可问题少，也要让追问链路真实。
- 可执行反馈优先于笼统评分：用户需要知道“哪里不好、为什么、怎么改”。
- 个性化优先于通用题库：问题要与简历、岗位和当前回答挂钩。
- 1.0 先闭环，后增强：先完成最小可用流程，再做 RAG、能力模型和代码面试。

## 核心流程
1. 用户填写目标岗位、面试类型、难度并上传简历。
2. 系统解析输入信息并生成本场面试计划。
3. AI 按阶段发起提问，并根据回答执行追问。
4. 面试结束后生成结构化报告。
5. 用户查看问题清单与建议，并选择开启下一轮针对性训练。

## 产品核心功能
- 面试流程模拟（最重要）。
  流程建议为：自我介绍 -> 项目深挖 -> 技术考察 -> 场景追问 -> 总结反馈。
- 个性化提问。
  问题生成依据：简历、岗位 JD、面试类型、用户回答上下文。
- 追问机制。
  当用户回答过浅、过泛、过于模板化或出现明显矛盾时，系统应自动追问。
- 面试反馈系统。
  反馈应包含：亮点、问题点、原因分析、改进建议、参考回答思路。

说明：原文中的“标准答案”建议改为“参考回答思路”。面试题通常不存在唯一标准答案，否则容易让用户误以为背模板即可通过面试。

### 追问机制示例
用户回答：我用 Redis 做缓存。
系统可继续追问：
- 为什么不用本地缓存？
- 缓存与数据库如何保证一致性？
- 高并发场景下如何避免缓存击穿或雪崩？
- 如果 Redis 故障，系统如何降级？

### 报告输出建议
- 总体评价。
- 维度评分：技术能力、表达能力、项目深度、稳定性。
- 代表性问题清单。
- 每个问题对应的改进建议。
- 推荐复训主题和下一步学习路径。

## 需求范围管理
### In-Scope：1.0 必做范围
- 模块一：面试创建。
  输入项建议为：目标岗位、面试类型、难度、目标公司（可选）、简历上传。
  输出为：生成本场面试配置与初始问题计划。
- 模块二：AI 文本面试会话。
  支持多轮提问、阶段切换、基于回答的自动追问、面试结束控制。
- 模块三：面试报告。
  支持总评、维度评分、问题诊断、改进建议和推荐训练方向。
- 模块四：历史记录与基础复盘。
  至少保留最近面试记录，便于用户查看上一场报告并再次训练。

### Limited Scope：1.0 可保留但先简化的能力
- 即时反馈建议默认关闭，仅在用户主动开启时提供简短提示，避免打断面试沉浸感。
- 问题权重先作为后台配置，不作为 1.0 的用户可配置项。
- 用户能力模型先通过“标签化弱点摘要”实现，不单独做复杂成长系统。

### Out-of-Scope：1.0 暂不实现
- 实时视频 / 语音面试。
- 多人群面或多面试官协同模式。
- 企业招聘系统与面试官后台。
- AI 自动生成简历。
- 在线手撕代码编辑器。
- 自动抓取公开面经或开放网页内容的 RAG 能力。

说明：RAG 建议延后。1.0 若一定要做知识增强，建议只支持用户上传 JD、简历和补充资料，不要先做开放式互联网资料聚合，避免数据质量和合规问题。

## 非功能性要求
- 单轮回复时延目标：普通问题 3-8 秒，可接受上限 10 秒。
- 简历支持格式：PDF、DOCX、TXT。
- 用户数据需支持删除，简历和面试记录需具备最小化存储策略。
- 评分与建议需输出可解释理由，避免只给分不说明。
- 对明显不安全、歧视性或不相关问题进行拦截。

## 风险与建议
- 评分可信度风险：如果分数来源不透明，用户会质疑结果。
- 追问质量风险：如果追问太模板化，真实感会明显下降。
- 范围膨胀风险：1.0 同时做 RAG、能力模型、代码面试会显著拉长周期。
- 用户预期风险：需要明确说明这是训练工具，不是实际录用结果预测工具。

## 建议的 1.0 MVP 边界
1. 必须做：面试创建、文本面试、多轮追问、结束报告。
2. 可以晚一点做：历史记录、二次训练、资料增强。
3. 明确不在首版做：视频语音、开放式 RAG、代码编辑器。
